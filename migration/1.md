## Migration Creation vs Execution Timeline

### Understanding the Two-Step Process

**Important:** There are TWO distinct steps that happen at DIFFERENT times:

#### Step 1: Creating the Migration File (Development Time)
```bash
npm run migration:create CreateOrderLevelBusinessReport
```

**What happens:**
- TypeORM CLI generates a new file with **current timestamp**
- Creates: `1725873212362-create_order_level_business_report.ts`
- Generates boilerplate class with empty `up()` and `down()` methods
- **File is created in your project directory**
- ⚠️ **Nothing is executed in the database yet!**

**Generated file:**
```typescript
export class createOrderLevelBusinessReport1725873212362 implements MigrationInterface {
    public async up(queryRunner: QueryRunner): Promise<void> {
        // You write your logic here
    }

    public async down(queryRunner: QueryRunner): Promise<void> {
        // You write rollback logic here
    }
}
```

#### Step 2: Writing Migration Logic
You manually write the SQL/code in the `up()` method:
```typescript
public async up(queryRunner: QueryRunner): Promise<void> {
    await queryRunner.query(`
        CREATE VIEW order_level_business_report AS
        SELECT order_date, seller_code, 
               COUNT(*) AS number_of_orders,
               SUM(product_price_subtotal) AS product_sales_amount
        FROM ec_sales_data 
        GROUP BY order_date, seller_code
    `);
}
```

#### Step 3: Running the Migration (Deployment Time)
```bash
npm run migration
```

**This is when** the migration actually executes against your database!

### Timeline Example

```
August 9, 2024 10:15 AM
  ↓
  Developer runs: npm run migration:create CreateOrderLevelBusinessReport
  ↓
  TypeORM creates file: 1725873212362-create_order_level_business_report.ts
  Timestamp: 1725873212362 = August 9, 2024, 10:15:42 AM
  ↓
  Developer writes SQL logic in up() method
  ↓
  Commits to Git
  ↓
  (Several days/weeks/months pass...)
  ↓
February 6, 2026 (TODAY!)
  ↓
  DevOps/Developer runs: npm run migration
  ↓
  Migration executes and creates the view in database
  ↓
  Record inserted in migrations table with timestamp 1725873212362
```

### What the Timestamp Represents

✅ **When the migration file was created** (in your codebase)  
✅ **Determines execution order** (older timestamps run first)  
❌ **NOT when it was executed** (that's tracked separately)  

**Think of it as:** The timestamp is the migration's "birthdate", not its "execution date"!

### TypeORM's Internal Algorithm

```typescript
// Pseudocode of what TypeORM does internally
async runMigrations() {
    // 1. Load all files matching the glob pattern
    const files = glob.sync(["./src/database/migration/*.ts"]);
    
    // 2. Import each migration class
    const migrations = files.map(file => require(file));
    
    // 3. Query the migrations table
    const executed = await query("SELECT * FROM migrations");
    
    // 4. Find pending migrations (not in database)
    const pending = migrations.filter(m => 
        !executed.some(e => e.name === m.name)
    );
    
    // 5. Sort by timestamp (chronological order)
    pending.sort((a, b) => a.timestamp - b.timestamp);
    
    // 6. Execute each pending migration
    for (const migration of pending) {
        await migration.up(queryRunner);
        
        // 7. Record execution
        await query(
            "INSERT INTO migrations (timestamp, name) VALUES ($1, $2)",
            [migration.timestamp, migration.name]
        );
    }
}
```

## Will Migrations Run Multiple Times?

### Short Answer: **NO** ✅

TypeORM's migration system is **idempotent** - safe to run multiple times!

### First Execution
```bash
npm run migration
```

**What happens:**
1. TypeORM checks the `migrations` table
2. Finds no entry for `createOrderLevelBusinessReportView1725873212362`
3. ✅ **Executes the migration** (runs `up()` method)
4. Creates the view in database
5. Inserts tracking record:
   ```sql
   INSERT INTO migrations (timestamp, name) 
   VALUES (1725873212362, 'createOrderLevelBusinessReportView1725873212362');
   ```

### Second Execution (and beyond)
```bash
npm run migration
```

**What happens:**
1. TypeORM checks the `migrations` table
2. ✅ **Finds the entry** for `createOrderLevelBusinessReportView1725873212362`
3. ❌ **Skips this migration** (already executed)
4. Console output: `No migrations are pending`

### The Migrations Table

TypeORM creates this table automatically:

```sql
CREATE TABLE migrations (
    id SERIAL PRIMARY KEY,
    timestamp BIGINT NOT NULL,
    name VARCHAR NOT NULL
);
```

**Example data after your migrations run:**

| id | timestamp | name |
|----|-----------|------|
| 1 | 1669698070631 | UserSeed1669698070631 |
| 2 | 1691557772140 | ecProductColumnRename1691557772140 |
| 3 | 1691741864094 | ecProductConsumpationTaxUpdate1691741864094 |
| 4 | 1692176204835 | createSellerMasterView1692176204835 |
| 5 | 1725873212362 | createOrderLevelBusinessReportView1725873212362 |

### To Re-run a Migration

If you need to run a migration again:

#### Option 1: Delete from migrations table
```sql
DELETE FROM migrations 
WHERE name = 'createOrderLevelBusinessReportView1725873212362';
```
Then run `npm run migration` again.

#### Option 2: Create a new migration (Recommended)
```bash
npm run migration:create UpdateOrderLevelBusinessReport
```

## Naming Conventions

### File Naming
```
{timestamp}-{description}.ts
```

**Examples:**
- `1725873212362-create_order_level_business_report.ts`
- `1691557772140-ec_product_column_rename.ts`

### Class Naming
```typescript
export class {DescriptionInCamelCase}{timestamp} implements MigrationInterface
```

**Examples:**
```typescript
export class createOrderLevelBusinessReportView1725873212362 implements MigrationInterface
export class ecProductColumnRename1691557772140 implements MigrationInterface
export class UserSeed1669698070631 implements MigrationInterface
```

### The Pattern

| Component | File Name | Class Name |
|-----------|-----------|------------|
| **Timestamp** | First | Last |
| **Description** | Last | First |
| **Separator** | Hyphen (`-`) | None (camelCase) |

**File Name Pattern:**
```
1725873212362-create_order_level_business_report.ts
└─────┬─────┘ └──────────┬───────────────────┘
  Timestamp          Description (snake_case)
```

**Class Name Pattern:**
```typescript
createOrderLevelBusinessReportView1725873212362
└──────────────┬──────────────────┘└─────┬─────┘
     Description (camelCase)         Timestamp
```

### Why This Matters

⚠️ **The class name MUST include the timestamp** - TypeORM uses this to:
1. Match the file to the class
2. Track execution in the migrations table
3. Determine execution order

## Quick Reference

### Create a Migration
```bash
npm run migration:create MyMigrationName
```

### Run Pending Migrations
```bash
npm run migration
```

### Check Which Migrations Ran
```sql
SELECT * FROM migrations ORDER BY timestamp;
```

### Migration File Requirements
✅ Class implements `MigrationInterface`  
✅ Class name includes timestamp  
✅ Has `up()` method (required)  
✅ Has `down()` method (for rollbacks)  
✅ File in `./src/database/migration/` directory  

### Best Practices
1. ✅ Always use `npm run migration:create` to generate files
2. ✅ Never modify timestamps after creation
3. ✅ Write reversible `down()` methods
4. ✅ Test migrations on development database first
5. ✅ Keep migrations small and focused
6. ✅ Never edit a migration after it's been run in production
