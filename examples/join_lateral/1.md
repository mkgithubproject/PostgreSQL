# 7️⃣ Why This Query Is Efficient

Execution behavior:

```
Filter orders
      ↓
For each order
      ↓
Index lookup in ec_order_product
      ↓
Aggregate items
      ↓
Final summary
```

So PostgreSQL **does NOT scan the whole `ec_order_product` table**.

Instead it does **small index lookups per order**.

---

💡 **One important thing:**
For this to be fast, you should have this index:

```sql
CREATE INDEX idx_order_product_lookup
ON ec_order_product (seller_code, order_id);
```

The confusion is normal because **`LATERAL` joins work differently from normal joins**.
Think of it like a **loop** where the right query runs **once for every row of the left table**.

I'll explain it visually step-by-step.

---

# 1️⃣ Your Query

```sql
FROM ec_order o

LEFT JOIN LATERAL (
    SELECT SUM(op.sales_volumn) AS items_sold
    FROM ec_order_product op
    WHERE op.order_id = o.order_id
      AND op.seller_code = o.seller_code
) items ON TRUE
```

Key point:

```text
items query can see o.order_id
```

This is **only possible because of `LATERAL`**.

---

# 2️⃣ Step 1 — Filter Orders

First PostgreSQL filters `ec_order`.

Example result:

| order_id | seller_code | amount |
| -------- | ----------- | ------ |
| 1        | m00000      | 100    |
| 2        | m00000      | 200    |

Think of this as:

```text
orders = [1,2]
```

---

# 3️⃣ Step 2 — PostgreSQL Loops Through Orders

Internally PostgreSQL behaves like this pseudo-code:

```text
for each row in ec_order:
    run the lateral query
```

---

# 4️⃣ First Loop (order_id = 1)

Postgres substitutes values into the lateral query:

```sql
SELECT SUM(op.sales_volumn)
FROM ec_order_product op
WHERE op.order_id = 1
AND op.seller_code = 'm00000'
```

Result:

```text
items_sold = 5
```

Join result becomes:

| order_id | amount | items_sold |
| -------- | ------ | ---------- |
| 1        | 100    | 5          |

---

# 5️⃣ Second Loop (order_id = 2)

Query becomes:

```sql
SELECT SUM(op.sales_volumn)
FROM ec_order_product op
WHERE op.order_id = 2
AND op.seller_code = 'm00000'
```

Result:

```text
items_sold = 5
```

Join result:

| order_id | amount | items_sold |
| -------- | ------ | ---------- |
| 2        | 200    | 5          |

---

# 6️⃣ Final Joined Table

After all loops finish:

| order_id | amount | items_sold |
| -------- | ------ | ---------- |
| 1        | 100    | 5          |
| 2        | 200    | 5          |

Then aggregation happens.

# 8️⃣ Why It Is Called LATERAL

Because the right query can **look sideways (laterally)** at columns of the left table.

Normal join **cannot do this**.

Example (invalid without LATERAL):

```sql
SELECT *
FROM ec_order o
JOIN (
   SELECT *
   FROM ec_order_product
   WHERE order_id = o.order_id
) t
```

`o.order_id` is **not visible** unless you use `LATERAL`.

---

# 9️⃣ Real PostgreSQL Execution

With an index on `(seller_code, order_id)` the engine does:

```text
Nested Loop
   -> scan filtered orders
   -> index lookup in ec_order_product
```

So instead of scanning the whole table:

```text
lookup order_id=1
lookup order_id=2
```

---

# 🔑 Simple Rule

Normal join:

```text
scan table A
scan table B
match rows
```

LATERAL join:

```text
scan table A
for each row in A:
    run query on table B
```
